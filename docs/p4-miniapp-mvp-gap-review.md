# P4 小程序 MVP 缺口复盘

记录时间：2026-05-28

## 结论

P4 已具备第一版 MVP 交易主链路的闭环验证条件，建议进入 P4 收尾验收，不在当前阶段展开完整地址模块。

理由：

- 地址模块会牵动用户地址表、订单地址快照、订单创建入参、后台订单展示和历史订单兼容，不适合作为 P4 收尾前的小补丁。
- 当前目标是验证私域商品交易闭环，订单备注和默认配送说明可以支撑本地 MVP 演示。
- 优惠券、积分、运费、售后、物流轨迹属于运营增强或履约增强，不应阻塞 P4 MVP。

## 已跑通能力

- 微信登录骨架和本地会话存储。
- 首页、分类页、商品详情页读取真实商品和分类接口。
- SKU 选择、加入购物车、购物车勾选和数量调整。
- 从购物车创建订单。
- 订单列表分页、下拉刷新、筛选、空态和加载到底提示。
- 订单详情状态展示、支付、取消、确认收货、错误态重试和返回。
- mock 支付回调推动订单进入待发货。
- 后台商品、订单、发货和统计基础能力。
- 本地种子数据、初始化脚本和交易链路 smoke 脚本。

## P4 收尾前保留缺口

| 缺口                 | 当前处理                                             | 建议阶段               |
| -------------------- | ---------------------------------------------------- | ---------------------- |
| 真实微信支付         | 当前使用 mock 支付和 mock notify 验证状态推进        | P5 联调上线准备        |
| 收货地址             | 订单确认页使用默认配送说明，提交后商家联系确认       | P5 或 P6 独立地址模块  |
| 优惠券 / 积分 / 运费 | MVP 不纳入交易主链路                                 | P6 运营增长            |
| 物流轨迹             | 后台只维护物流公司和单号，小程序展示基础物流信息     | P5/P6 履约增强         |
| 售后退款             | 后端已有部分状态基础，小程序未做完整售后入口         | P6 售后模块            |
| 小程序构建           | 当前沙箱下 Taro 构建存在 macOS 底层 panic / 挂起风险 | 本机开发环境或 CI 复验 |

## 本节点体验处理

- 订单确认页不再展示“后续地址模块接入”这类开发占位文案。
- 个人中心的“收货地址”和“优惠券”入口标注为待开放，并提供点击 toast 反馈，避免用户误以为页面卡住。

## 下一步建议

进入 P4 MVP 收尾验收：

- 对照 MVP 主链路验收清单做一次文档化检查。
- 在 Docker / MySQL / Redis / Server 可用环境下执行 `pnpm smoke:transaction`。
- 复跑 `pnpm typecheck`、`pnpm format:check`。
- 小程序 build 放到非沙箱本机或 CI 环境复验。
